## 系统概述

这是一个基于 ScriptableObject 的**解耦事件系统**，实现了观察者模式（Observer Pattern）。它允许不同的游戏对象之间进行通信，而无需直接引用彼此。

### 核心思想
使用 ScriptableObject 作为"事件通道"，发送者和接收者都只需要引用这个通道，而不需要知道对方的存在。

---

## 三个核心组件

### 1. GameEvent.cs - 事件中心（ScriptableObject）
**角色定位：** 事件通道 / 消息总线

**职责：**
- 维护监听器列表
- 提供注册/注销监听器的接口
- 触发事件并通知所有监听器

**类比：** 就像一个广播电台，负责管理订阅者并发送广播

---

### 2. GameEventListener.cs - 事件监听器（MonoBehaviour）
**角色定位：** 事件接收者

**职责：**
- 挂载到需要响应事件的 GameObject 上
- 在启用时注册到 GameEvent
- 在禁用时从 GameEvent 注销
- 接收事件并执行响应操作

**类比：** 就像收音机，调到某个频道（GameEvent）并播放收到的内容（response）

---

### 3. GameEventTrigger.cs - 事件触发器（MonoBehaviour）
**角色定位：** 事件发送者

**职责：**
- 挂载到需要触发事件的 GameObject 上
- 提供触发事件的方法
- 支持延迟触发和自动触发

**类比：** 就像广播电台的播音员，负责按下"发送"按钮

---

## 架构设计模式

### 观察者模式（Observer Pattern）
```
┌─────────────────────────────────────────────────────────┐
│                    GameEvent (Subject)                   │
│                  [ScriptableObject]                      │
│  ┌────────────────────────────────────────────────┐    │
│  │  - listeners: List<GameEventListener>          │    │
│  │  + Raise()                                     │    │
│  │  + RegisterListener(listener)                  │    │
│  │  + UnregisterListener(listener)                │    │
│  └────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
           ▲                                    │
           │ 注册                               │ 通知
           │                                    ▼
┌──────────────────────┐           ┌──────────────────────┐
│ GameEventListener    │           │ GameEventTrigger     │
│  [MonoBehaviour]     │           │  [MonoBehaviour]     │
│                      │           │                      │
│  - gameEvent         │           │  - eventToTrigger    │
│  - response          │           │  + TriggerEvent()    │
│  + OnEventRaised()   │           └──────────────────────┘
└──────────────────────┘
```

### 解耦优势
- **发送者** 不需要知道谁在监听
- **接收者** 不需要知道谁发送的
- **双方** 只需要知道 GameEvent 这个"中间人"

## 工作流程

### 完整生命周期

```
1. 创建阶段
   └─> 在 Unity 编辑器中创建 GameEvent ScriptableObject 资源
       （右键 -> Create -> Tutorial/03_Advanced/Game Event）

2. 配置阶段
   ├─> 在监听器 GameObject 上添加 GameEventListener 组件
   │   ├─> 拖入 GameEvent 资源到 gameEvent 字段
   │   └─> 配置 response（UnityEvent）要执行的操作
   │
   └─> 在触发器 GameObject 上添加 GameEventTrigger 组件
       └─> 拖入同一个 GameEvent 资源到 eventToTrigger 字段

3. 运行时阶段
   ├─> 监听器启用（OnEnable）
   │   └─> 自动注册到 GameEvent 的监听器列表
   │
   ├─> 触发器调用 TriggerEvent()
   │   └─> GameEvent.Raise() 被调用
   │       └─> 遍历所有监听器
   │           └─> 调用每个监听器的 OnEventRaised()
   │               └─> 执行 response.Invoke()
   │
   └─> 监听器禁用（OnDisable）
       └─> 自动从 GameEvent 注销
```

## 代码逻辑详解

### GameEvent.cs 核心逻辑

#### 1. 监听器管理
```csharp
private List<GameEventListener> listeners = new List<GameEventListener>();
```
- 使用 List 存储所有注册的监听器
- 这个列表在运行时动态维护

#### 2. 注册监听器
```csharp
public void RegisterListener(GameEventListener listener)
{
    if (!listeners.Contains(listener))
    {
        listeners.Add(listener);
    }
}
```
**逻辑：**
- 检查监听器是否已存在（避免重复注册）
- 不存在则添加到列表

**调用时机：** GameEventListener 的 OnEnable() 中自动调用

#### 3. 注销监听器
```csharp
public void UnregisterListener(GameEventListener listener)
{
    if (listeners.Contains(listener))
    {
        listeners.Remove(listener);
    }
}
```
**逻辑：**
- 检查监听器是否存在
- 存在则从列表中移除

**调用时机：** GameEventListener 的 OnDisable() 中自动调用

**为什么需要注销？**
- 防止内存泄漏
- 避免调用已销毁对象的方法
- 保持监听器列表干净

#### 4. 触发事件（核心方法）
```csharp
public void Raise()
{
    // 从后向前遍历
    for (int i = listeners.Count - 1; i >= 0; i--)
    {
        if (listeners[i] != null)
        {
            listeners[i].OnEventRaised();
        }
    }
}
```
**为什么从后向前遍历？**
- 如果在事件响应中有监听器注销自己，从前向后遍历会导致索引错乱
- 从后向前遍历即使移除元素也不会影响未遍历的元素索引

**空值检查：**
- `if (listeners[i] != null)` 防止调用已销毁的 GameObject 上的组件

---

### GameEventListener.cs 核心逻辑

#### 1. 自动注册/注销机制
```csharp
private void OnEnable()
{
    if (gameEvent != null)
    {
        gameEvent.RegisterListener(this);
    }
}

private void OnDisable()
{
    if (gameEvent != null)
    {
        gameEvent.UnregisterListener(this);
    }
}
```
**为什么使用 OnEnable/OnDisable？**
- `OnEnable`: GameObject 激活时调用（包括场景加载、SetActive(true)）
- `OnDisable`: GameObject 禁用时调用（包括场景卸载、SetActive(false)、销毁）
- 这确保了监听器的生命周期与 GameObject 同步

**好处：**
- 自动化管理，无需手动调用
- 避免忘记注销导致的问题
- 支持动态启用/禁用 GameObject

#### 2. 事件响应
```csharp
public void OnEventRaised()
{
    response?.Invoke();
}
```
**逻辑：**
- `response` 是一个 UnityEvent，可以在 Inspector 中配置
- `?.` 是空值条件运算符，如果 response 为 null 则不执行
- `Invoke()` 触发 UnityEvent 中配置的所有方法

**UnityEvent 的优势：**
- 可视化配置，无需写代码
- 支持多个响应方法
- 支持传递参数（如果使用 UnityEvent<T>）

---

### GameEventTrigger.cs 核心逻辑

#### 1. 触发事件的主方法
```csharp
public void TriggerEvent()
{
    if (eventToTrigger == null)
    {
        Debug.LogWarning("未分配 GameEvent！");
        return;
    }

    if (delayTime > 0)
    {
        Invoke(nameof(TriggerEventInternal), delayTime);
    }
    else
    {
        TriggerEventInternal();
    }
}
```
**逻辑流程：**
1. 检查是否分配了 GameEvent（防御性编程）
2. 如果设置了延迟时间，使用 `Invoke` 延迟调用
3. 否则立即调用内部触发方法

#### 2. 内部触发方法
```csharp
private void TriggerEventInternal()
{
    eventToTrigger.Raise();
    Debug.Log($"触发了事件: {eventToTrigger.name}");
}
```
**职责：**
- 调用 GameEvent 的 Raise() 方法
- 输出日志便于调试

#### 3. 自动触发
```csharp
void Start()
{
    if (triggerOnStart)
    {
        TriggerEvent();
    }
}
```
**用途：**
- 场景加载时自动触发事件
- 适合初始化场景、播放开场动画等场景

## 使用场景示例

### 场景 1：玩家死亡事件

**设置：**
1. 创建 `PlayerDiedEvent` (GameEvent)
2. 在 UI 管理器上添加 GameEventListener
   - gameEvent: PlayerDiedEvent
   - response: 显示游戏结束界面
3. 在玩家脚本中添加 GameEventTrigger
   - eventToTrigger: PlayerDiedEvent

**代码示例：**
```csharp
// 在玩家脚本中
public class Player : MonoBehaviour
{
    public GameEvent playerDiedEvent;

    void TakeDamage(float damage)
    {
        health -= damage;
        if (health <= 0)
        {
            playerDiedEvent.Raise(); // 触发死亡事件
        }
    }
}
```

**优势：**
- UI 管理器不需要引用玩家对象
- 玩家不需要知道 UI 的存在
- 可以轻松添加更多响应（音效、粒子效果等）

---

### 场景 2：关卡完成事件

**设置：**
1. 创建 `LevelCompleteEvent` (GameEvent)
2. 多个监听器：
   - 成就系统：解锁成就
   - 存档系统：保存进度
   - UI 系统：显示胜利界面
   - 音频系统：播放胜利音乐
3. 在关卡管理器中触发

**一对多通信：**
```
LevelCompleteEvent.Raise()
    ├─> 成就系统响应
    ├─> 存档系统响应
    ├─> UI 系统响应
    └─> 音频系统响应
```

---

### 场景 3：按钮触发事件

**设置：**
1. 创建 `ButtonClickEvent` (GameEvent)
2. 在按钮上添加 GameEventTrigger
3. 在 Button 组件的 OnClick 中调用 `TriggerEvent()`

**Inspector 配置：**
```
Button (UI)
├─> Button Component
│   └─> OnClick()
│       └─> GameEventTrigger.TriggerEvent()
└─> GameEventTrigger Component
    └─> eventToTrigger: ButtonClickEvent
```

## 优势分析

### 1. 解耦（Decoupling）
- 发送者和接收者完全独立
- 修改一方不影响另一方
- 易于维护和扩展

### 2. 可复用性
- GameEvent 可以在多个场景中使用
- 同一个事件可以有多个监听器
- 同一个监听器可以监听多个事件

### 3. 可视化配置
- 在 Inspector 中拖拽配置
- 无需修改代码即可改变行为
- 设计师友好

### 4. 易于测试
- 可以在编辑器中右键测试
- GameEvent 提供了 `[ContextMenu]` 测试方法
- 不需要运行整个游戏

### 5. 性能优势
- ScriptableObject 是单例，内存占用小
- 避免了 SendMessage 的性能开销
- 避免了 Find 查找对象的开销

### 6. 灵活性
- 支持延迟触发
- 支持自动触发
- 支持运行时动态添加/移除监听器

## 扩展建议

### 1. 带参数的事件
可以创建泛型版本传递数据：
```csharp
public class GameEvent<T> : ScriptableObject
{
    public void Raise(T value) { ... }
}
```

### 2. 事件优先级
为监听器添加优先级，控制响应顺序

### 3. 事件历史
记录事件触发历史，便于调试

### 4. 条件触发
添加条件判断，只在满足条件时触发

---

## 总结

这个事件系统是 Unity 中实现解耦通信的优雅方案：

- **GameEvent** = 通信通道（ScriptableObject）
- **GameEventListener** = 接收者（挂在响应对象上）
- **GameEventTrigger** = 发送者（挂在触发对象上）

通过这种设计，你可以构建灵活、可维护的游戏系统，而不会陷入复杂的对象引用网络中。
